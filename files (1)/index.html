Unnamed Pipes
Unidirectional byte-stream channel between parent and child processes via pipe()
How It Works
A pipe is a kernel buffer with two file descriptors: fd[0] (read end) and fd[1] (write end). After fork(), both parent and child inherit both ends. The unused end must be closed to avoid blocking.
pipe() syscall fork() + exec() Parent ↔ Child Kernel Buffer
Characteristics
DirectionUnidirectional (one way)
ScopeRelated processes only
PersistenceDies with process
Throughput~113 MB/s
LatencyLow (kernel copy)

ipc_core.c — pipe demo
void demo_unnamed_pipe(void) {
  int pipefd[2];
  pid_t pid;
  pipe(pipefd);          // creates read[0] and write[1]

  pid = fork();
  if (pid == 0) {        // CHILD — reader
    close(pipefd[1]);    // close unused write end
    read(pipefd[0], buf, sizeof(buf));
    close(pipefd[0]);
    exit(EXIT_SUCCESS);
  } else {               // PARENT — writer
    close(pipefd[0]);    // close unused read end
    write(pipefd[1], msg, strlen(msg));
    close(pipefd[1]);    // signals EOF to child
    wait(NULL);
  }
}
Named Pipes (FIFO)
Filesystem-backed pipes enabling communication between unrelated processes via mkfifo()
How It Works
A FIFO appears as a special file in the filesystem at a known path (e.g. /tmp/my_fifo). Any process with the right permissions can open it — no shared ancestry required. Data flows FIFO order through the kernel buffer.
mkfifo() syscall Filesystem path Unrelated processes Blocking open()

// Writer process (can be completely unrelated)
int fd = open("/tmp/ipc_sim_fifo", O_WRONLY);
write(fd, msg, strlen(msg));
close(fd);

int fd = open("/tmp/ipc_sim_fifo", O_RDONLY);
read(fd, buf, sizeof(buf));
close(fd);
unlink("/tmp/ipc_sim_fifo"); // cleanup
Shared Memory
Fastest IPC — multiple processes map the same physical memory pages via shmget() / shmat()
How It Works
The kernel allocates a memory segment identified by a key. Processes attach it into their own address space with shmat(). Data written by one process is immediately visible to all others — no copying. Requires synchronisation to prevent race conditions.
shmget / shmat Needs semaphores Zero-copy Fastest IPC
Shared Segment Layout
counter int32 — synchronised with semaphore
message char[256] — payload buffer
ready int32 — completion flag
Source Code
ipc_core.c — shared memory
// Allocate shared memory segment
int shmid = shmget(SHM_KEY, SHM_SIZE, IPC_CREAT | 0666);
SharedData *shdata = (SharedData *)shmat(shmid, NULL, 0);

// Writer (child) — under semaphore protection
sem_wait(semid);
shdata->counter++;
snprintf(shdata->message, sizeof(shdata->message),
         "Write complete! counter=%d", shdata->counter);
shdata->ready = 1;
sem_post(semid);

// Cleanup — critical to avoid resource leaks
shmdt(shdata);
shmctl(shmid, IPC_RMID, NULL);
semctl(semid, 0, IPC_RMID);
Semaphores
Synchronisation primitive — prevents race conditions in shared memory access
How It Works
A semaphore is a kernel counter. sem_wait() decrements it (blocks if 0). sem_post() increments it (wakes waiting processes). A binary semaphore (value 0/1) acts as a mutex lock for critical sections.
Semaphore state machine
val = 1 → Resource free, sem_wait() proceeds
val = 0 → Locked, caller blocks
sem_post → Releases lock, unblocks waiter

Race Condition — Code Comparison
UNSAFE — data race
// No protection — WRONG
for (int i = 0; i < ITERS; i++) {
    (*counter)++;  // ← RACE HERE
    // read-modify-write NOT atomic
    // another process can interrupt
}
SAFE — semaphore lock
// Semaphore-protected — CORRECT
for (int i = 0; i < ITERS; i++) {
    sem_wait(semid);  // lock
    (*counter)++;     // ← safe
    sem_post(semid);  // unlock
}
IPC Concepts
Core theory behind inter-process communication in modern operating systems
🔗
Process Isolation
Each process has its own virtual address space. The OS kernel prevents direct memory access between processes. IPC mechanisms are the controlled channels through which data can cross these boundaries safely.
Virtual MemoryMMUKernel
⚙️
Concurrency
Multiple processes run simultaneously on the CPU. The scheduler interleaves their execution. Without coordination, shared state becomes inconsistent — this is the root of all race conditions in multiprocess systems.
SchedulingContext SwitchParallelism
🔒
Mutual Exclusion
A guarantee that only one process executes a critical section at a time. Achieved via semaphores, mutexes, or spinlocks. Prevents data corruption when multiple processes access shared resources.
SemaphoreMutexSpinlock
🔄
Deadlock
A circular wait where process A holds resource X waiting for Y, while process B holds Y waiting for X. Both block forever. Prevented by lock ordering, timeouts, or the Banker's Algorithm.
Coffman ConditionsBanker's Alg
⚡
Race Condition
A bug where program outcome depends on the non-deterministic interleaving of concurrent operations. Classic example: two processes reading, modifying, and writing a shared counter — increments get lost.
Non-deterministicData Loss
📬
Producer-Consumer
A classic IPC pattern: one process produces data, another consumes it via a shared buffer. Semaphores coordinate access: one counts empty slots, another counts full slots. Pipes implement this natively.
BufferSemaphoresBlocking
